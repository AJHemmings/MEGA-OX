## What This Project Is

MEGA-OX is a React + TypeScript implementation of Ultimate Naughts and Crosses (a 3×3 grid of tic-tac-toe boards). Currently it has:
- Single-player vs a random-move AI (no real intelligence)
- Local 2-player (same screen, same mouse)
- A working, clean OOP game engine in `src/models/Game.ts` — fully decoupled from React
- Stubbed online multiplayer UI (disabled buttons in `MultiplayerMenu.tsx`)

---

## What Was Designed This Session

A complete network multiplayer platform. The full design is in:
`docs/plans/2026-03-16-network-multiplayer-design.md`

**Summary of what gets built:**
- Supabase backend (auth, database, realtime, storage)
- Email/password + Google OAuth authentication
- User profiles with username, avatar, rank tier, stats, match history
- Live network multiplayer via Supabase Realtime (DB-authoritative, not broadcast)
- Friendly games (invite code or open matchmaking)
- Season mode (monthly rolling, matchmade by MMR)
- Tournament mode (admin-created, bracket-based, registration required)
- Hidden MMR (Elo) system — 7 visible rank tiers (Novice → Grand Master)
- Leaderboard (top 1000, rank tier shown, raw MMR never exposed)
- Cash shop infrastructure (coins, transactions, cosmetic items)
- Cosmetics system (marker skins, board skins, themes — bought or won)
- Login streak gamification with daily rewards
- News system (admin-posted, slideshow on main menu)
- Tutorial overlay system (per-page, first visit only)
- Full main menu redesign (lobby-style UI)
- 20 database tables + 1 Postgres view

## Key Decisions Made & Why

| Decision | Rationale |
|---|---|
| Supabase as sole backend | PostgreSQL needed for relational data (rankings, friends, history). Realtime built in. Free tier sufficient. New account to avoid existing project limits. |
| DB-authoritative realtime (Approach B) | Postgres RLS enforces game rules server-side. No state divergence. Match history automatic. Future features (replay, spectate) need the move log. |
| Hidden MMR + visible rank tier | Industry standard. Players see a title, not a number. Prevents obsessing over exact ELO. Raw MMR blocked by RLS — never reaches client. |
| "Season" not "League" | Avoid football connotations. Seasons feel like a time-limited competitive period, which is what they are. |
| New players start at 1000 MMR | Places them at top of Challenger tier — immediately feels like there's room to grow. |
| Matchmaking widening | ±200 MMR initially, widens every 30s to avoid players waiting forever. |
| `rules_config` jsonb on seasons | Season-specific matchmaking rules TBD — jsonb column means we can define them later without a migration. |
| Stripe payment ID in transactions | Monetization not built yet, but the column is there. Plugs in cleanly when ready. |
| Supabase MCP at implementation time | Allows Claude to directly manage DB schema, RLS policies, and storage from within the dev session. |
| No Vercel API routes | No need. Supabase handles auth, logic (via RLS), realtime, and storage. Adding Vercel as a middleman adds complexity for no gain. |
